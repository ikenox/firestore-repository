# Pipeline Query — spec-coverage gap analysis

> Work-list derived **from the type/runtime definitions**, not from memory. For
> every "special consideration" the types or the runtime deliberately encode on
> an IMPLEMENTED pipeline stage, this document checks whether the live
> behavioural spec
> (`packages/firestore-repository/src/__test__/pipeline-spec.ts`) has a test that
> actually asserts THAT rule. A stage being merely _exercised_ is not the same as
> a rule being _pinned_ — those get flagged as GAPs.
>
> Sources cross-referenced:
> `pipelines/pipeline.ts`, `pipelines/selection.ts`, `pipelines/expression.ts`,
> `pipelines/stage.ts`, `pipelines/source.ts`, and the research docs
> (`pipeline-query-{aggregate,unnest,identity,ordering,null-semantics,projection}-research.md`).
>
> Legend for the "covered?" column: **YES** = a live test asserts the rule;
> **GAP** = no live test pins it; **TYPE-ONLY** = the rule is a compile-time
> guard (a dotted output collapses the parameter to `never`), so it cannot be
> asserted by a live test at all — it belongs in the type-test suite
> (`selection.test.ts` / `expression.test.ts`), and is listed here only for
> completeness.

Implemented stages (from `docs/plan/pipeline-query.md` status table): `collection`
/ `collectionGroup`, `where`, `sort`, `limit`, `offset`, `select`, `addFields`,
`removeFields`, `aggregate`, `distinct`, `unnest`. Stubs (`fullReplaceWith`,
`mergeWith`, `union`, `findNearest`, `let`, `search`, `sample`, `database`,
`documents`, `literals`, DML) are out of scope.

## GAP SUMMARY (fill work-list)

Flat list of every missing LIVE test, ordered by stage. The output-name
uniqueness rule (a compile-time guard, not a live test) is separated below.

- **sort — null-vs-absent are distinct sort positions (null < absent < values):** the seed only has present/absent for `profile.gender`; add a field seeded with an actual `null` value to pin `null < absent` ascending and its mirror descending. Matters because the total order around absence is a documented divergence from core-query `orderBy`.
- **addFields — conditionality (optional-ancestor leaf becomes optional/nullable):** no addFields test derives a field from a path crossing an optional map; `WithConditionality` is shared with select but its addFields path is unexercised. Matters because addFields' schema claim would silently diverge from the row shape if the shared operator regressed for the addFields entry point.
- **aggregate — bare `Field` un-aliased group (`field('g')`):** only bare-string and aliased-expression groups are tested for aggregate (distinct does cover the bare-Field group). Matters for parity of the three accepted group forms.
- **aggregate — `arrayAggDistinct`:** the accumulator is never exercised anywhere; needs one live evaluation (distinct-collection + null/absent behaviour). Matters because its wire translation and semantics are entirely unpinned.
- **aggregate — `sum` result KIND (int64 vs float64 via `NumericResult`):** no `type()` oracle over `sum`; current sums are whole ints so the two kinds are never distinguished. Matters because the descriptor claim drives row decode and a wrong kind ships silently.
- **aggregate — `average` result KIND (always float64):** never pinned via `type()`.
- **aggregate — `count`/`countAll`/`countDistinct`/`countIf` KIND (int64):** never pinned via `type()`.
- **aggregate — `minimum`/`maximum`/`first`/`last` KIND (operand's own kind):** never pinned via `type()`.

### OUTPUT-NAME UNIQUENESS (resolved — type-level rejection)

The backend's "OUTPUT NAMES MUST BE UNIQUE — no overlap wins, ever" rule
(aggregate-research §"Output shape restrictions") is enforced as a COMPILE-time
guard (ban-what-silently-succeeds). No live test can exist for it — the offending
query does not type-check — so it is pinned by negative type tests, not spec
tests.

One rule, one guard: the aggregate/distinct output-name SET — every group output
name UNION every accumulator alias — must be pairwise distinct, with NO
resolution. `selection.ts:UniqueAggregateOutputs` maps over a groups/accumulators
tuple and collapses any element whose output name occurs a second time in the
combined name list to `never`; it is applied as a parameter intersection on
`Pipeline.aggregate` / `Pipeline.distinct` (the `UndottedGroupAliases`
precedent). Each accumulator alias is checked against the combined
groups-∪-accumulators set and each group name against the groups alone, so all
three collision kinds reject and a cross-collision surfaces once (on the
accumulator side).

- **accumulator alias == group name** — rejected at the type level.
  `AggregateSchema` (`MergeSchemas<AccumulatorSchema, GroupSchema>`) now only ever
  joins a DISJOINT set, so the merge is a plain union with no winner.
- **duplicate accumulator alias** — rejected at the type level.
  `AccumulatorSchema` folds a disjoint record (`OverwriteMerge` never overwrites).
- **duplicate GROUP output name** (both `aggregate` and `distinct`) — rejected at
  the type level. The group list is carried into the stage node as-is: the
  aggregate-family group list no longer runs `dropOverriddenSelections` (the
  runtime does not resolve overlaps the type forbids). `select` / `addFields` /
  `unnest` KEEP last-wins / added-field-wins — the backend accepts overlaps
  there, so overlap resolution is stage-specific.

Coverage: `pipeline.test.ts` `aggregate/distinct output names must be pairwise
unique: every collision is rejected` pins all three collision kinds (both stages)
as `@ts-expect-error` plus positive controls. `selection.test.ts` no longer
asserts any accumulator-wins / group last-wins collapse for these stages (those
described rejected queries).
